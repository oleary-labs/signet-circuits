# Toolchain management

## Why pin so aggressively

`nargo` compiles Noir source to ACIR. `bb` takes ACIR + witness and produces a proof and a verification key. `bb.js` does the same thing in the browser (and in Node). `noir_js` does witness generation in JS.

Versions across these tools do **not** cleanly interoperate:

- Different `bb` versions can produce different VKs from the same ACIR.
- A proof generated by `bb vX` may not verify under the VK generated by `bb vY`.
- `bb` and `bb.js` must match on the proving-system version for proofs to be interchangeable.
- `nargo` determines the ACIR format; `bb` must understand it.

So for any single Signet release, there is exactly one compatible combination of these four versions. `toolchain.json` is the file that records that combination.

## The one file

```json
{
  "nargo":        "1.0.0-beta.X",
  "bb":           "0.X.Y",
  "bb_js":        "0.X.Y",
  "noir_js":      "1.0.0-beta.X",
  "noir_jwt_rev": "<commit-or-tag>"
}
```

Everything else reads from here:

| Location | How it uses `toolchain.json` |
|---|---|
| `Dockerfile` | Build args `NARGO_VERSION` and `BB_VERSION`, set from the file in `.github/workflows/release.yml`. |
| `scripts/install-toolchain.sh` | Reads with `jq`, invokes `noirup -v` and `bbup -v`. |
| `scripts/build.sh` | Asserts installed versions match, then compiles. Copies the whole file into `metadata.json`. |
| `packages/ts/` | Imports `metadata.json` at build time, exports `TOOLCHAIN` constant. Peer-deps on `@aztec/bb.js` and `@noir-lang/noir_js` match the same file. |
| `packages/go/` | Same idea — embeds `metadata.json`, exposes `Toolchain` struct via `JWTMetadata()`. |

## The Docker image flow

The Dockerfile bakes a pinned `nargo` and `bb` into an image that gets published on release as `ghcr.io/signet-protocol/signet-circuits-toolchain:<version>`.

The bundler's Dockerfile then does:

```dockerfile
FROM ghcr.io/signet-protocol/signet-circuits-toolchain:1.2.3 AS toolchain
FROM golang:1.22 AS build
# ... build the Go bundler binary ...

FROM toolchain AS runtime
COPY --from=build /bundler /usr/local/bin/bundler
CMD ["/usr/local/bin/bundler"]
```

Now the bundler's runtime has exactly the `bb` version the artifacts were built against. There is no way for the bundler's `bb` to drift from the `signet-circuits` release it depends on — they come from the same container image.

Bumping `bb` in prod is then just bumping the base image tag and the Go module version together. Both come from the same `signet-circuits` release.

## Runtime assertions

Both consumer packages surface a startup assertion:

```ts
// UI
import { assertBbJsVersion } from "@signet/circuits";
await assertBbJsVersion();
```

```go
// Bundler
if err := circuits.AssertToolchain(); err != nil {
    log.Fatalf("toolchain mismatch: %v", err)
}
```

These fail fast if the host environment's `bb` / `bb.js` version doesn't match the pinned version embedded in the consumer package. Version skew becomes a crash at startup, not a silent verification failure at proof time.

## What a toolchain bump looks like

1. Open a PR on `signet-circuits` editing `toolchain.json` (and `.tool-versions` to match).
2. CI rebuilds artifacts, runs smoke tests against the new toolchain. If anything fails (including cross-verify), PR is blocked.
3. Merge, then cut a release tag. CI publishes the npm package, Go module, and Docker image.
4. Open PRs in UI, bundler, and protocol that bump the `signet-circuits` dependency version. Each one is a one-line change plus whatever the `CHANGELOG.md` flagged as breaking.
5. Deploy in lockstep. Until everyone is on the new version, keep the old container images available for rollback.

## Common traps

- **Using a branch instead of a rev for `noir-jwt`.** Never do this. The whole point of pinning is reproducibility.
- **Letting `.tool-versions` drift from `toolchain.json`.** CI should grep and fail if they disagree. (TODO: add this check.)
- **Copying artifacts out of a running container into another image without carrying `metadata.json`.** The metadata is the proof that the artifacts are consistent. If you strip it, you've lost the ability to assert at startup.
- **Installing `bb` globally from `bbup` latest without `-v`.** Easy mistake in local dev. The install script is the safer path.
