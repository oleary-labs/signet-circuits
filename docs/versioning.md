# Versioning and release flow

## Versioning policy

Semantic versioning, interpreted strictly for this repo:

- **Major (`X.0.0`)** — breaking circuit changes. The VK has changed; old proofs will not verify. Any downstream deployment holding old VKs must redeploy.
- **Minor (`1.X.0`)** — any toolchain bump (`nargo`, `bb`, `bb.js`, or `noir_js` version change), or additive circuit features that generate new VKs. Toolchain bumps are always at least minor, because proofs from the old version may not verify under the new VK.
- **Patch (`1.2.X`)** — no artifact changes. Doc updates, build-script tweaks, CI changes, refactors to `zkbench`. If `artifacts/**` hashes to the same value before and after the change, it's a patch.

In practice almost every release is minor or major. Patches exist mostly for in-repo tooling.

## What gets released together

A single `vX.Y.Z` tag triggers `.github/workflows/release.yml`, which produces:

- `@signet/circuits@X.Y.Z` on npm
- `github.com/signet-protocol/signet-circuits/packages/go @ vX.Y.Z` (via the `packages/go/vX.Y.Z` tag)
- `ghcr.io/signet-protocol/signet-circuits-toolchain:X.Y.Z` Docker image
- Uploaded `artifacts/` zip on the GitHub release page (optional; useful for manual inspection)

All four artifacts encode the same `toolchain.json` and the same circuit hash. Mixing versions across them is not supported — a bundler using `packages/go@v1.2.3` must run in a container based on `signet-circuits-toolchain:1.2.3`.

## Release flow

1. **Open a release PR.** Bump `CHANGELOG.md` under `[Unreleased]` with notes; call out toolchain changes explicitly. Do not edit version numbers in `packages/ts/package.json` — CI does that.
2. **Merge to `main`.** CI runs build + smoke. Green = ready to tag.
3. **Tag.** `git tag vX.Y.Z && git push origin vX.Y.Z`. `release.yml` takes over.
4. **Verify publish.** Confirm the npm package, Docker image, and Go module tag all resolved. `release.yml` fails the job on any failure, but spot-check manually for the first few releases while the workflow is young.
5. **Open downstream bump PRs.** UI, bundler, and any other consumer each get a one-line dependency bump. Link each PR back to the `signet-circuits` release notes.
6. **Deploy in lockstep.** If the bump involves a toolchain change, the bundler must redeploy (new base image) and the UI must ship the new bb.js at the same time. Don't half-deploy — proofs generated by the old bundler will fail to verify against the new UI's VK (and vice versa).

## Rollback

If a release turns out bad:

- **Do not delete the release tag or npm version.** Deletion breaks consumers that already pulled it.
- Tag a new patch or minor release with the fix.
- If the bad release shipped a broken artifact (rare — smoke tests should catch this), publish a deprecation note on the npm version (`npm deprecate @signet/circuits@X.Y.Z "reason"`).
- Downstream consumers pin to the last known good version and redeploy.
